# KZ Car Market — antifraud data pipeline (kolesa.kz, Алматы)

Учебно-боевой DS-проект: ежедневный сбор объявлений о продаже авто,
чистка, детекция аномальных/мошеннических объявлений, разметка,
подготовка данных к ML (модель цены, классификатор фрода).

## Архитектура (DAG)

```
parser.py ──► raw_ads          (паспорта объявлений, самодозаполняются)
          ──► sightings        (журнал: кого видели в какой день)
          ──► photos           (все фото, полный размер, long format)
check_status.py ──► ad_status      (active / archived / deleted)
clean.py  ◄── raw_ads + sightings + ad_status + enriched + manual_labels.csv
          ──► clean_data          (флаги is_suspicious + причины)
enrich.py ◄── clean_data (очередь: подозрительные первыми)
          ──► enriched         (растаможка, руль, привод, комментарий продавца)
photo_dedup.py ◄── photos + clean_data (очередь: подозрительные первыми)
          ──► photo_hashes, photo_duplicates (одно фото — разные машины)
explore.py ──► dashboard.png, suspicious_sorted.csv, labeling_queue.csv
```

Слои (medallion): **raw** неизменяем (append-only), **clean** пересобирается
из raw в любой момент, артефакты EDA — производные.

Хранилище — Postgres (`docker-compose.yaml`), не CSV. Идёт пилот
дозаписи (см. «Статус миграции на Postgres» ниже): `parser.py`,
`check_status.py`, `enrich.py` пока продолжают писать CSV как раньше
(источник истины на время пилота) и ДОПОЛНИТЕЛЬНО пишут в Postgres;
`clean.py`/`enrich.py`/`photo_dedup.py`/`explore.py` уже читают только
из Postgres. `manual_labels.csv` остаётся файлом навсегда — его правит
человек руками.

## Airflow (демо-слой, портфолио — НЕ путь по умолчанию)

Тот же DAG, что и `run_all.py`, но через Apache Airflow — показать
оркестрацию по-взрослому (UI, отдельный retry на таск, backfill).
Основной ежедневный сбор как был — `run_all.py` + cron, этот блок
поднимается только явно и не мешает основному:

```bash
docker compose --profile airflow up -d --build   # первый раз — соберёт образ (~5-10 мин)
```

UI: http://localhost:8080, логин `admin` / `admin` (создаётся один раз
в `airflow-init`). DAG `kolesa_antifraud_pipeline` — лежит на паузе
после первого деплоя, включи вручную в UI (переключатель слева от
имени DAG'а), если хочешь, чтобы он реально стартовал по расписанию
`0 9 * * *`.

**Важно**: `docker compose --profile airflow down` останавливает **все**
контейнеры проекта, включая основной `postgres` — профили в Compose
фильтруют, что поднимается по `up`, но не что попадает под `down`.
Чтобы погасить только Airflow, не трогая основную БД:
```bash
docker compose stop airflow-webserver airflow-scheduler airflow-postgres airflow-init
```

Проверено вживую: образ собирается (constraint-конфликт Airflow/SQLAlchemy
решён — Airflow ставит свою 1.4.54, `db.py` использует только базовые
API, совместимые и с 1.4, и с 2.0), DAG грузится без import errors,
таск `explore` реально выполнился внутри контейнера (подключился к
`postgres` по имени сервиса, а не `localhost` — важно для
container-to-container сети) и записал `dashboard.png` на хост через
общий volume.

```bash

```

## Справочник команд (шпаргалка)

**Что гоняешь регулярно (точки входа):**

| команда | что делает |
|---|---|
| `python run_all.py` | полный цикл: листинг → статусы → обогащение ∥ фото → чистка → отчёт |
| `python run_all.py --light` | только новый листинг + пересборка флагов; per-ad сеть (статусы/обогащение/фото) НЕ трогает — их отдаём `catch_up`. Легче, быстрее, не долбит kolesa мимо бюджета |
| `python run_all.py --fast` | совсем без сети: пересобрать `clean`+EDA из уже собранного raw |
| `python catch_up.py` | показать пробелы (статусы/обогащение/avgPrice+бейдж/фото) + остаток дневной квоты, спросить — добирать ли |
| `python catch_up.py --run` | добрать по одной порции на джоб (в пределах суточного бюджета) |
| `python catch_up.py --run --until-done` | использовать всю оставшуюся дневную квоту (round-robin по джобам); стоп при исчерпании бюджета / 429 |
| `python pipeline_status.py` | «пульт»: бэклоги, покрытие, ETA — офлайн, без запросов |
| `python evaluate_detector.py` | precision/recall детектора против ручных вердиктов (после разметки) |
| `python -m pytest test_pipeline.py -q` | тесты (каждый = застывший реальный баг) |

**Отдельные джобы — обычно НЕ вручную** (их оркестрирует `run_all`/`catch_up`,
и порядок важен: `clean` идёт до `enrich` — строит очередь приоритетов — и
после — вливает обогащение). Знать полезно для отладки:

| команда | флаг/настройка | что делает | хост |
|---|---|---|---|
| `python parser.py` | env `KOLESA_MAX_PAGES` (глубина листания на сегмент; дефолт 10 ≈800/день) | собрать листинг → raw/sightings/photos | kolesa.kz |
| `python check_status.py` | — | статусы исчезнувших: active/archived/deleted | kolesa.kz |
| `python enrich.py` | — (до 20/прогон) | обогатить страницы: растаможка, привод, комментарий, avgPrice, бейдж | kolesa.kz |
| `python backfill_avgprice.py` | `--all` (иначе ~20) | дозаполнить avgPrice+бейдж у старых enriched | kolesa.kz |
| `python photo_dedup.py` | — (до 300/прогон) | pHash фоток: «одно фото — разные машины» | CDN kcdn.kz |
| `python clean.py` | — | пересобрать `clean_data` (флаги, robust z-score) из raw+enriched | офлайн |
| `python explore.py` | — | дашборд + `suspicious_sorted.csv` + `labeling_queue.csv` | офлайн |
| `python migrate_to_postgres.py` | — | разово перелить CSV → Postgres (идемпотентно) | офлайн |

### Запуск — руками, а не по крону

Проект живёт на домашнем ноуте, который большую часть суток спит/выключен.
Обычный `cron` пропущенные запуски НЕ навёрстывает (в 9:00 машина спит →
джоб просто не выполнился; `launchd` на ноуте — так же). Поэтому запуск
**ручной**: сам гоняешь `run_all.py` / `catch_up.py`, когда за ноутом. Это
безопасно — пайплайн полностью резюмируемый (raw append-only, сентинелы,
суточный бюджет), пропущенный день ничего не ломает: добьёшь на следующем
заходе. На always-on сервере (VPS/облако) это же вешается на cron/
systemd-timer — там осмысленно; на ноуте — нет.

Типичный день во время добора backlog: `python run_all.py --light` (быстро
собрать новьё) + `python catch_up.py --run --until-done` (добрать пробелы на
дневную квоту). Оба безопасны по бану (см. ниже) и резюмируемы.

### Анти-бан: суточный бюджет запросов на хост

Бан ловится по ОБЪЁМУ запросов с одного IP за сутки, а не по паузе между
ними (паузы 4-8с уже внутри каждого сетевого джоба). Поэтому `catch_up.py`
держит общий на хост дневной потолок числа запросов (`DAILY_BUDGET`:
kolesa 200 — общий на три kolesa-джоба, это один IP; CDN 1200 — отдельный
хост). Плюс сами порции мелкие (по 20 запросов у kolesa-джобов) — меньше
«всплеск». Счётчик — `logs/.catch_up_budget.json` (сбрасывается с новыми
сутками). Как только квота хоста выбрана, его джобы встают до завтра.
Так ни один запуск — даже случайный ручной, даже `--until-done` — не
пробьёт суточный объём. `--until-done` использует всю оставшуюся квоту
разом (round-robin по джобам), обычный `--run` — по одной порции на джоб.
Backlog рассасывается за ~2-3 недели на безопасной скорости.

(История: 2026-07-23 домашний IP словил временный бан kolesa при ~270
запросах/сутки с одного IP — бюджет и размеры порций с тех пор снижены.)

Важно про двойную нагрузку: суточный бюджет считает только запросы
`catch_up`. Если в тот же день гнать ПОЛНЫЙ `run_all` (он сам делает
статусы+обогащение ~270 запросов к kolesa), суммарный объём на IP =
run_all + бюджет catch_up. Поэтому во время добора backlog запускай
`run_all --light` (без per-ad сети) — тогда весь per-ad трафик идёт через
один бюджетный `catch_up`, и потолок реально соблюдается.

## Логика детектора (clean.py)

1. Нормализация типов, возраст, индикаторы пропусков (пропуск пробега —
   MNAR: сам факт информативен).
2. Жёсткие правила: физически невозможные значения (год, цена, пробег,
   пробег/возраст, «б/у с нулевым пробегом»).
3. Статистика: модифицированный z-score (медиана + MAD, порог 3.5) по
   log-цене в иерархических группах модель → бренд → возраст.
   Дешёвый хвост = подозрение, дорогой = информационная пометка.
4. Оправдания (exculpation): слова об убитости в описании/комментарии,
   «Растаможен: Нет», ручной вердикт `legit` — снимают флаг с пометкой
   `low_price_explained`.
5. Усиления: `cheap_and_urgent` (аномально дёшево + «срочно»),
   ручной вердикт `fraud`.
6. Дубликаты-перезаливы (кроме дилерских новых машин).
7. `shared_photo_diff_car` (из `photo_dedup.py`): одно и то же фото
   (perceptual hash, ТОЧНОЕ совпадение) встречается у объявлений с
   РАЗНОЙ маркой/моделью, годом (±2) или ценой (>15%) — сигнал
   ворованного/переиспользованного фото, который не видит ни цена, ни
   текст. Порог откалиброван на реальных парах: hamming 2-4 = одна
   дилерская студия/разные машины (ложные), hamming 0 = настоящий дубль.

Оправдания-исключения (флаг снимается или понижается до `info_flags`):
`low_price_explained` (повреждения с учётом ОТРИЦАНИЙ — «нет гнили» ≠
«гниль»), `dealer_financing_price` (в цене взнос, не стоимость),
`kolesa_price_ok` (кросс-чек: наша цена в пределах рынка по независимому
эталону kolesa — см. ниже), `repost_unconfirmed` (совпадение
title+год+цена без второго фактора). Лексикон повреждений — в `damage.py`
(единый источник, поиск с окном отрицаний).

**Кросс-чек средней ценой kolesa.** kolesa на странице объявления сама
считает среднюю рыночную цену (`avgPrice` в embedded JSON) — сравнивая
внутри точного год+поколение+привод+двигатель+кузов+КПП+растаможка и
ИСКЛЮЧАЯ выбросы. Это чище нашей грубой корзины «модель × 0-3 года»
(которая мешает модельные годы). `enrich.py` вытаскивает его бесплатно
(страница и так качается), `backfill_avgprice.py` дозаполняет старые
строки. В `clean.py`: если наш z-score флагует «дёшево», но цена в
пределах ~20% от kolesa-эталона — наш флаг ложный, снимаем
(`kolesa_price_ok`). **Только кросс-чек/валидатор, НЕ признак модели
цены** — иначе модель училась бы копировать kolesa (target leakage).

## Тесты

```bash
python -m pytest test_pipeline.py -q     # 27 тестов, миллисекунды
```
Тестируются ЧИСТЫЕ функции (вход → выход, без сети и файлов), и в
частности — **сама логика детектора**, а не только парсинг:
- `test_negated_damage_not_detected` / `test_real_damage_still_detected`
  — фродовая/честная строка → флаг поднимается/снимается корректно;
- `test_repost_confirmed_by_mileage` / `_unconfirmed_goes_to_info_only`
  — двухуровневый перезалив;
- `test_exact_hash_diff_model` / `_near_hash_not_flagged_studio_lookalike`
  — фото-детектор ловит дубль, не ловит студийного двойника;
- `test_robust_z_ignores_single_outlier` — z-score не маскируется выбросом;
- `test_confusion_matrix_counts` — харнесс метрик считает верно.

Почти каждый тест — застывший реальный баг (см. git log), чтобы
регрессия в правиле не пропустила фрод молча.

## Качество детектора (метрики)

`python evaluate_detector.py` считает precision/recall/F1 флага
`is_suspicious` против ручных вердиктов (`data/manual_labels.csv`) +
матрицу ошибок + **precision каждого правила отдельно** (какой флаг
ловит фрод, какой красит честных). Пока вердиктов нет — печатает, что
разметить; метрика появится в тот же миг, когда появится разметка.
Это следующий по важности шаг: **без ground truth мы знаем, ЧТО
детектор флагует, но не знаем, ПРАВ ли он.**

## Статус миграции на Postgres

Схема (`sql/init/01_schema.sql`, авто-применяется на чистом volume) и
разовый бэкфилл (`migrate_to_postgres.py`) готовы, применены
(2026-07-18). Сейчас идёт пилот дозаписи: `parser.py`,
`check_status.py`, `enrich.py` продолжают писать CSV как единственный
источник истины и ДОПОЛНИТЕЛЬНО (best-effort, сбой не роняет прогон)
пишут в Postgres. `clean.py`, `enrich.py` (выбор очереди), `photo_dedup.py`,
`explore.py` уже читают только из Postgres.

**Жёсткий срок cutover (не «когда-нибудь»):** два источника истины —
техдолг, который живёт вечно, если не поставить дату. Критерий и срок:
**после 7 подряд чистых ежедневных прогонов** (счётчики CSV↔Postgres
совпадают, проверяется `migrate_to_postgres.py` в режиме сверки) —
убрать CSV-запись из parser/check_status/enrich. Дедлайн: **до
2026-08-05**. Если к этой дате не сделано — CSV остаётся авторитетным,
а Postgres-дозапись выпиливается как незавершённый эксперимент (лучше
один честный источник, чем два вечных). `clean_data.csv`-снапшот
остаётся навсегда (дёшево). Откат кода — через git (первый коммит —
снапшот до миграции). CSV-файлы raw-слоя не удаляются никогда.

## Ручная разметка (human-in-the-loop)

`explore.py` создаёт `data/eda/labeling_queue.csv` — открой, пройди по
URL, заполни `verdict` (legit / fraud / unknown) и `comment`, затем
скопируй строки в `data/manual_labels.csv`. Следующий `clean.py` учтёт
вердикты.
Разметка копит обучающую выборку для будущего классификатора.

## Известные ограничения (честно)

- «Продано» определяется по архивации страницы — прокси, момент известен
  с точностью до интервала (интервальная цензурированность).
- Комментарии на казахском языке лексиконом не покрываются.
- Залоги/аресты/юридический статус — вне наших данных в принципе.
- photos_count из листинга — нижняя граница (сайт отдаёт ≤5 превью).
- Пробег отсутствует у 20–40% объявлений в самом источнике (MNAR,
  измерено 37% на 2026-07-18); доля растёт с ростом базы — не
  зашивайте одно число в код, ориентируйтесь на `is_mileage_missing`.
- **Нет метрик качества, пока пуст `manual_labels.csv`**: детектор
  флагует по эвристикам, но precision/recall не измерены — харнесс
